Basics

What is a compiler?
Answer: A compiler is a program that translates source code written in a high-level programming language into target code (machine code, assembly, or another high-level language), typically producing an executable or intermediate representation.

What is the difference between a compiler and an interpreter?
Answer: A compiler translates the entire program ahead-of-time into target code and then execution runs that output; an interpreter directly executes source (or an intermediate form) statement-by-statement without producing a separate executable.

What are the phases of a compiler?
Answer: Typical phases are: lexical analysis, syntax analysis (parsing), semantic analysis, intermediate code generation, optimization, and target code generation (and optionally linking).

What is a source program?
Answer: The source program is the original program written by the programmer in a source language (e.g., C, Java).

What is a target program?
Answer: The target program is the output of the compiler—code in the target language (machine code, assembly, or another language) that can be executed on the target machine.

What is cross-compilation?
Answer: Cross-compilation is compiling code on one machine/architecture to produce executable code for a different target machine/architecture.

What is bootstrapping?
Answer: Bootstrapping is the process of writing a compiler for a language in that same language and progressively compiling improved versions, or the technique of building a compiler using earlier compiler versions.

Lexical Analysis

What is lexical analysis?
Answer: Lexical analysis is the compiler phase that reads the source characters and groups them into tokens (identifiers, keywords, literals, operators) and removes whitespace/comments.

What is a token?
Answer: A token is a categorized atomic unit of the source language (e.g., identifier, number, keyword) produced by the lexer; it has a token type and often an associated lexeme or attribute.

What is a lexeme?
Answer: A lexeme is the actual character sequence in the source that matches a token’s pattern (e.g., the string "while" is a lexeme for the keyword token while).

What is a pattern?
Answer: A pattern is a rule (usually expressed as a regular expression) that describes all lexemes belonging to a particular token class.

What is a lexical analyzer?
Answer: A lexical analyzer (lexer/scanner) is the program/module that implements lexical analysis: it reads characters and outputs tokens to the parser.

What are the functions of a lexical analyzer?
Answer: Functions include: tokenizing input, removing whitespace/comments, recognizing lexemes, returning token types and attributes, and handling lexical errors.

What is the difference between tokens and lexemes?
Answer: Tokens are abstract categories (types); lexemes are concrete character sequences that match a token’s pattern.

Why are regular expressions used in lexical analysis?
Answer: Regular expressions precisely and compactly specify token patterns; they map directly to finite automata used by lexers for efficient pattern matching.

What are keywords and identifiers?
Answer: Keywords are reserved words with special meaning in the language (e.g., if, while); identifiers are programmer-defined names for variables, functions, etc.

Syntax Analysis

What is parsing?
Answer: Parsing is the process of analyzing a sequence of tokens to determine its grammatical structure according to a formal grammar, producing a parse tree or derivation.

What is a parser?
Answer: A parser is the compiler component that performs syntax analysis, checking token sequences against the grammar and constructing parse/syntax trees.

What is syntax analysis?
Answer: Syntax analysis is the compiler phase that verifies program structure (syntax) using a context-free grammar and produces hierarchical representations (parse trees).

What are context-free grammars?
Answer: Context-free grammars (CFGs) are formal grammars with production rules of the form A -> α where A is a nonterminal and α is a string of terminals/nonterminals; they describe programming language syntax.

What is a derivation?
Answer: A derivation is a sequence of production applications that rewrites the start symbol into a string of terminals (the input), showing how the string is generated by the grammar.

What is a parse tree?
Answer: A parse tree is a tree representation of a derivation where internal nodes are nonterminals, leaves are terminals, and edges show applied productions.

What is ambiguity in grammar?
Answer: A grammar is ambiguous if some string in the language has more than one distinct parse tree (or derivation) under that grammar.

What is leftmost derivation?
Answer: A leftmost derivation is a derivation where at each step the leftmost nonterminal is expanded.

What is rightmost derivation?
Answer: A rightmost derivation expands the rightmost nonterminal at each step.

Grammar

What is a grammar?
Answer: A grammar is a formal description of a language consisting of terminals, nonterminals, a start symbol, and production rules that define how strings are formed.

What are terminals and non-terminals?
Answer: Terminals are the basic symbols of the language (tokens); non-terminals are syntactic variables used in productions to group language constructs.

What is left recursion?
Answer: Left recursion occurs when a nonterminal A has a production A -> Aα (direct) or A -> ... -> Aα (indirect), causing leftmost expansion to loop in top-down parsers.

Why must left recursion be removed in LL(1) parsing?
Answer: LL(1) (top-down) parsers use recursive-descent strategies; left recursion leads to infinite recursion, so it must be eliminated or transformed.

What is left factoring?
Answer: Left factoring is a grammar transformation that refactors productions with common prefixes so the parser can decide which production to use by looking ahead.

Why is left factoring required?
Answer: It’s required to make the grammar suitable for predictive parsing by removing common prefixes that prevent a single-token lookahead decision.

What is an ambiguous grammar?
Answer: An ambiguous grammar allows multiple parse trees for the same input string, making parsing and meaning (e.g., operator associativity) unclear.

Give an example of an ambiguous grammar.
Answer: E -> E + E | E * E | ( E ) | id is ambiguous because expressions like id + id * id can have multiple parse trees without precedence rules.

FIRST and FOLLOW

What is the FIRST set?
Answer: FIRST(X) is the set of terminals that can begin strings derived from symbol X (terminal or nonterminal); for productions producing epsilon, epsilon may be in FIRST.

What is the FOLLOW set?
Answer: FOLLOW(A) is the set of terminals that can appear immediately to the right of nonterminal A in some sentential form; for the start symbol, $ (end marker) is included.

Why are FIRST and FOLLOW sets required?
Answer: They are used to construct predictive parsing tables (LL(1)) and to decide which production to apply based on lookahead tokens.

What symbol is placed in FOLLOW of the start symbol?
Answer: The end-of-input marker, conventionally `$` (or EOF), is placed in FOLLOW(start-symbol).

What is ε (epsilon)?
Answer: Epsilon denotes the empty string (a production that yields no terminals) used to represent nullable productions.

What is the role of epsilon in FIRST and FOLLOW?
Answer: If a nonterminal can derive epsilon, then epsilon is included in FIRST; FOLLOW computation uses epsilon to propagate lookahead to earlier symbols and include FOLLOW of the left-hand nonterminal.

LL(1) Parsing

What is LL(1) parsing?
Answer: LL(1) parsing is a top-down predictive parsing method that reads input Left-to-right, produces a Leftmost derivation, and uses 1 token of lookahead.

What does LL(1) stand for?
Answer: Left-to-right scan of input (L), Leftmost derivation (L), 1-token lookahead (1).

What is a predictive parser?
Answer: A predictive parser is an LL parser that uses a parsing table (built from FIRST/FOLLOW) to decide productions without backtracking.

What is an LL(1) parsing table?
Answer: It's a table indexed by nonterminals and lookahead terminals indicating which production to apply for a given pair; built using FIRST and FOLLOW sets.

How is an LL(1) table constructed?
Answer: For each production A -> α, add A->α into table[A,a] for every a in FIRST(α); if epsilon in FIRST(α), add A->α into table[A,b] for every b in FOLLOW(A); handle conflicts to detect non-LL(1).

What are the advantages of LL(1) parsing?
Answer: Simplicity, fast O(n) parsing, easy error reporting and recovery, and straightforward implementation as recursive-descent.

What are the limitations of LL(1) parsing?
Answer: Limited language class (cannot handle left recursion or certain ambiguities), requires grammar transformations (left-factoring), and struggles with constructs needing more lookahead.

Bottom-Up Parsing

What is bottom-up parsing?
Answer: Bottom-up parsing builds a parse by starting from the input terminals and reducing them to higher-level constructs until the start symbol is produced; it effectively reverses a rightmost derivation.

What is shift-reduce parsing?
Answer: Shift-reduce parsing is a bottom-up technique that alternates shifting tokens onto a stack and reducing the top of the stack by grammar productions when a handle is found.

What is handle pruning?
Answer: Handle pruning is the act of identifying a handle (the right substring corresponding to a production RHS) and reducing it to the LHS, thereby shortening the right-sentential form toward the start symbol.

What is a handle?
Answer: A handle is a substring of the current sentential form that matches the right-hand side of a production and whose reduction corresponds to one step in the rightmost derivation.

What is a viable prefix?
Answer: A viable prefix is a prefix of a right-sentential form that can appear on the parser stack during a shift-reduce parse; it does not extend beyond the right end of the handle.

What is reduction?
Answer: Reduction replaces a sequence of symbols matching a production RHS on the parse stack with the production’s LHS, representing a reverse derivation step.

What is acceptance in parsing?
Answer: Acceptance occurs when the parser has reduced the input to the start symbol and the input is fully consumed (often stack contains start symbol and lookahead is $).

LR Parsing

What is LR parsing?
Answer: LR parsing is a family of bottom-up parsing algorithms that are efficient and can handle a large class of grammars; they scan Left-to-right and produce a Rightmost derivation in reverse.

What does LR stand for?
Answer: Left-to-right scan of input (L) and Rightmost derivation in reverse (R).

What is LR(0)?
Answer: LR(0) is the simplest LR parser using items without lookahead; parsing decisions are based solely on state and stack, which is often insufficient for many grammars.

What is SLR(1)?
Answer: SLR(1) (Simple LR) augments LR(0) with 1-token lookahead using FOLLOW sets to resolve reduce actions; more powerful than LR(0) but weaker than canonical LR(1).

What is CLR(1)?
Answer: CLR(1) (Canonical LR(1)) uses full LR(1) items with explicit one-token lookahead for each item, yielding the most precise parse tables for 1-token lookahead.

What is LALR(1)?
Answer: LALR(1) (Lookahead LR) merges LR(1) states with identical cores to reduce table size while often preserving CLR(1) behaviour; commonly used in parser generators.

What is an LR parsing table?
Answer: An LR table has ACTION (shift/reduce/accept) entries for terminals and GOTO entries for nonterminals indexed by parser states; it's used by the LR parsing algorithm.

What are ACTION and GOTO tables?
Answer: ACTION[state,terminal] tells the parser to shift, reduce by a production, or accept; GOTO[state,nonterminal] tells the state to transition to after a reduction.

What is a shift action?
Answer: Shift pushes the current lookahead token onto the stack and transitions to a new state, consuming the token.

What is a reduce action?
Answer: Reduce replaces the symbols on the stack that match a production RHS with the production LHS and jumps to the GOTO state based on the new top.

What is an accept action?
Answer: Accept indicates successful parsing—input is syntactically valid and reduced to the start symbol with no remaining input.

What is a parser state?
Answer: A parser state encodes where the parser is in the grammar (often represented by a set of LR items) and determines possible actions on lookahead.

Lex and Yacc

What is Lex?
Answer: Lex (or flex) is a lexical analyzer generator that converts regular-expression rules into a scanner producing tokens for a parser.

What is Yacc?
Answer: Yacc (Yet Another Compiler Compiler, or bison) is a parser generator that consumes a grammar specification and produces a parser (usually LALR(1)).

What is the purpose of Lex?
Answer: Lex automates writing tokenizers by generating efficient C code (or other languages) that recognizes patterns and returns tokens to the parser.

What is the purpose of Yacc?
Answer: Yacc automates building parsers from grammar specs, generating parsing tables and driver code that invokes semantic actions.

What files are generated by Lex?
Answer: Lex typically generates a C source file (e.g., lex.yy.c) implementing the scanner.

What is yytext?
Answer: `yytext` is the global (or provided) character pointer/string in Lex that holds the current lexeme matched by the scanner.

What is yylex()?
Answer: `yylex()` is the scanner function generated by Lex that returns successive tokens (usually as integer codes) and fills attributes like `yylval`.

What is yyparse()?
Answer: `yyparse()` is the parser function generated by Yacc that drives parsing, calling `yylex()` for tokens and invoking semantic actions.

How do Lex and Yacc work together?
Answer: Lex provides tokens and lexical attributes to Yacc: the lexer (`yylex`) returns token codes and values to the parser (`yyparse`), which then applies grammar rules and actions.

Errors

What is lexical error?
Answer: A lexical error occurs when the lexer encounters a character sequence that does not match any token pattern (invalid token).

What is syntax error?
Answer: A syntax error occurs when the sequence of tokens does not conform to the grammar—parser cannot find a valid production to continue.

What is semantic error?
Answer: A semantic error refers to meaning-level problems (type mismatches, undeclared identifiers, incompatible operations) detected in semantic analysis.

What is error recovery?
Answer: Error recovery are strategies compilers use to continue processing after detecting an error to find more errors or to gracefully fail, instead of halting immediately.

What are the types of error recovery techniques?
Answer: Techniques include panic-mode (skip tokens until synchronization), phrase-level recovery (insert/delete tokens), error productions in grammar, and global correction (minimum edits), each with trade-offs.

Intermediate Code Generation

What is intermediate code?
Answer: Intermediate code is a low-level, machine-independent representation of the program (e.g., three-address code) produced between parsing and target code generation.

Why is intermediate code generated?
Answer: It decouples front-end language specifics from back-end code generation, facilitates optimization, and eases retargeting to different machines.

What is three-address code?
Answer: Three-address code (TAC) is a sequence of instructions where each instruction has at most three addresses/operands (e.g., t1 = t2 + t3), convenient for optimizations.

What are quadruples?
Answer: Quadruples represent TAC as 4-tuples (op, arg1, arg2, result), a common intermediate representation format.

What are triples?
Answer: Triples are another IR form that represent instructions as (op, arg1, arg2) with results referenced by instruction index rather than a named temporary.

Code Optimization

What is code optimization?
Answer: Code optimization transforms intermediate or target code to improve performance, reduce size, or lower resource usage while preserving semantics.

Why is optimization required?
Answer: To produce faster, smaller, or more power-efficient executables and to meet performance/resource constraints of applications and platforms.

What is constant folding?
Answer: Constant folding evaluates constant expressions at compile time (e.g., replace 2+3 with 5) to eliminate runtime computation.

What is dead code elimination?
Answer: Dead code elimination removes code that cannot be executed or whose results are never used, reducing size and improving performance.

What is common subexpression elimination?
Answer: It detects repeated expressions whose operands haven't changed and computes them once, reusing the result to avoid redundant computation.

Symbol Table

What is a symbol table?
Answer: A symbol table is a data structure used by the compiler to record information about identifiers (names): types, scopes, memory locations, attributes, and linkage.

What information is stored in a symbol table?
Answer: Typical entries include identifier name, type, scope level, memory address/offset, attributes (const/volatile), parameter info, and possibly links to AST nodes.

Which compiler phases use the symbol table?
Answer: Many phases use it: lexical (to categorize identifiers), semantic analysis (type checking, scope), intermediate code generation and code generation (addresses/offsets), and optimization.

Frequently Asked Viva Questions

Why are regular expressions used in lexical analysis but not syntax analysis?
Answer: Regular expressions define regular languages recognizable by finite automata (suitable for lexing). Syntax requires context-free grammars (nested structures) that need pushdown automata (parsers).

Can every grammar be converted to LL(1)?
Answer: No. Not every grammar can be transformed into LL(1) form; some languages inherently need more lookahead or are more naturally parsed bottom-up.

Which is more powerful: LL(1) or LR(0)?
Answer: LR(0) and LL(1) are different classes; generally LR(0) can handle some constructs LL(1) cannot, but canonical LR(1) is strictly more powerful than LL(1). Comparing LL(1) to LR(0), LR(0) handles fewer grammars than full LR(1) but the bottom-up family (LR) is more powerful overall.

Why is LR parsing preferred over LL parsing?
Answer: LR parsing can handle a much larger class of grammars (including most programming language constructs) without grammar transformations and handles ambiguity resolution (like precedence) better.

What is the difference between top-down and bottom-up parsing?
Answer: Top-down (e.g., LL) starts from the start symbol and expands productions to match input (predictive), while bottom-up (e.g., LR) reduces input tokens to nonterminals aiming to reach the start symbol.

Why is left recursion a problem?
Answer: Left recursion in a grammar causes recursive-descent top-down parsers to loop infinitely when expanding left-recursive productions; it must be removed for LL parsers.

What is the difference between a parse tree and a syntax tree?
Answer: A parse tree is a full representation of the grammar derivation including all terminals and nonterminals; a syntax tree (abstract syntax tree) is a simplified tree that omits redundant grammar nodes and focuses on program structure and semantics.

Which compiler phase is the most important and why?
Answer: All phases are important, but semantic analysis is critical because it enforces language rules (type safety, scope), ensures correctness, and provides information (types, symbol table) essential for code generation and optimization.
